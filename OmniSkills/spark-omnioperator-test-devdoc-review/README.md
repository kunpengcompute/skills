# spark-omnioperator-test-devdoc-review

审查 Spark OMNI 优化算子的开发设计文档，确保文档为下游 agent 自动生成测试设计提供足够的信息密度。

## 功能

- 审查设计文档的功能完整性（F1~F9 共 9 个维度）
- 评估行为可测试性（NULL/边界/异常/并发行为描述是否足够具体）
- 检查 SQL 示例多样性（类型覆盖、场景覆盖）
- 检查兼容性覆盖（OS/JDK/机型/Spark版本/存储格式）
- 生成结构化检视报告，包含判定、依据和修复建议

## 前置依赖

本 skill 需要用户提供待检视的开发设计文档（Markdown 格式）。文档应包含算子的设计流程、数据类型支持、开关配置、观测算子、SQL 用例示例等内容。

## 使用方式

向 AI 提供算子名称及设计文档路径即可触发，例如：

- "请检视 TableWrite 算子的设计文档，路径是 `/path/to/TableWrite_Design_Doc.md`"
- "帮我审查 HashJoin 的开发设计文档"
- "评审 Sort 算子的需求设计文档"

## 检视维度

### 功能完整性（影响最终结论）

| 维度 | 名称 | 说明 |
|:---|:---|:---|
| F1 | 测试约束 | 测试约束是否明确具体 |
| F2 | 数据类型覆盖 | 19 种数据类型是否逐项明确支持/不支持 |
| F3 | 开关配置 | 开关配置是否完整描述 |
| F4 | 观测算子 | 观测算子的 6 项信息是否齐全 |
| F5 | 设计流程 | 主流程、异常分支、流程图是否具备 |
| F6 | SQL 用例示例 | SQL 示例是否包含全部 9 项必要信息 |
| F7 | 兼容性 | 5 个兼容性维度是否全部覆盖 |

### 测试生成就绪度（影响最终结论）

| 维度 | 名称 | 说明 |
|:---|:---|:---|
| F8 | 行为可测试性 | 行为描述是否足够具体，agent 能否据此生成测试用例 |
| F9 | SQL 示例多样性 | SQL 示例是否覆盖多种场景（正常/边界/回退/组合等） |

### 判定逻辑

```
F1~F9 中任意功能性检视项为 ❌ → 最终结论 = ❌ 不通过
F1~F9 全部通过 → 最终结论 = ✅ 通过
非功能性问题不影响最终结论
```

## 输出

| 文件/产出 | 格式 | 描述 |
|:---|:---|:---|
| BigData_Spark_Operator_Test_{OperatorName}_Design_Document_Review_Report.md | Markdown | 结构化检视报告，包含各维度判定、依据和修复建议 |
| 最终交付总结 | 对话输出 | 检视结论、各维度结果、必须修复项、输出文件路径等汇总信息 |
