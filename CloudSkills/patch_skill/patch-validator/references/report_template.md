# 补丁校验报告模板

本文档提供如何解读校验报告的指南，以及常见的误差模式和排查方法。

## 报告结构

校验报告包含以下主要部分：

### 1. 执行摘要

执行摘要提供整体统计数据：

- **总补丁数**: 被校验的补丁文件数量
- **总 Hunks 数**: 所有补丁中的 hunk 总数
- **无误差 Hunks**: 可以精确应用的 hunk 数量和百分比
- **存在误差 Hunks**: 有轻微偏移但仍可应用的 hunk 数量
- **错误 Hunks**: 无法应用的 hunk 数量

### 2. 误差 Hunks 详情（如有）

列出所有存在轻微偏差的 hunks，包括：
- 偏移行数
- 相似度分数
- 原始行号 vs 实际匹配行号

### 3. 错误 Hunks 详情（如有）

详细列出所有无法匹配的 hunks，包括：
- 错误原因
- 上下文预览
- 修复建议

### 4. 详细结果

逐个补丁的详细分析，每个补丁包含：
- 提交信息（主题、哈希、作者、日期）
- 每个文件的校验结果
- 每个 hunk 的状态表格

---

## 状态说明

| 状态 | 图标 | 含义 | 操作建议 |
|------|------|------|----------|
| CLEAN | ✅ | 无误差 | 无需操作，补丁可以精确应用 |
| VARIATION | ⚠️ | 存在误差 | 检查偏移是否可接受，通常可以正常应用 |
| FAILED | ❌ | 错误 | 需要手动修复或调整 |

## 常见误差模式

### 1. 行号偏移 (Line Offset)

**现象**: VARIATION 状态，显示上下文偏移了 N 行

**原因**:
- 目标分支有额外的提交引入了新行
- 其他补丁先应用导致行号变化

**解决方案**:
- 通常 `git apply` 可以自动处理
- 如果偏移较大，检查是否有遗漏的依赖补丁

### 2. 空白字符变化 (Whitespace Changes)

**现象**: VARIATION 状态，相似度接近但非 100%

**原因**:
- Tab vs 空格差异
- 行尾空白字符差异
- CRLF vs LF 换行符差异

**解决方案**:
- 使用 `git apply --ignore-space-change`
- 使用 `git apply --ignore-whitespace`

### 3. 代码冲突 (Code Conflict)

**现象**: FAILED 状态，相似度低

**原因**:
- 目标代码已被其他补丁修改
- 补丁基于过时的代码版本
- 同一位置有不同的修改

**解决方案**:
- 手动合并冲突
- 使用 `git apply --3way` 尝试三方合并
- 重新生成补丁

### 4. 文件不存在 (File Not Found)

**现象**: FAILED 状态，"Target file not found"

**原因**:
- 文件在目标分支中不存在
- 文件路径错误
- 文件已被重命名或删除

**解决方案**:
- 检查目标分支是否正确
- 确认文件是否需要先由其他补丁创建
- 检查是否有重命名补丁需要先应用

## 排查失败 Hunks 的方法

### 步骤 1: 检查目标文件

```bash
# 检查文件是否存在
git show <branch>:path/to/file

# 查看文件内容
git show <branch>:path/to/file | head -n <line_number>
```

### 步骤 2: 手动尝试应用

```bash
# 尝试应用单个补丁
git apply --check path/to/patch.patch

# 查看详细错误信息
git apply --verbose path/to/patch.patch

# 尝试三方合并
git apply --3way path/to/patch.patch
```

### 步骤 3: 查找相似代码

```bash
# 在目标文件中搜索上下文
git show <branch>:path/to/file | grep -n "context string"
```

### 步骤 4: 比较差异

```bash
# 比较源分支和目标分支的文件
git diff <source_branch> <target_branch> -- path/to/file
```

## 最佳实践

### 补丁生成

```bash
# 生成标准格式的补丁
git format-patch -o patches/ origin/main..HEAD

# 生成带更多上下文的补丁
git format-patch -U5 -o patches/ origin/main..HEAD
```

### 补丁应用顺序

1. 先应用基础补丁（无依赖）
2. 按依赖关系顺序应用
3. 最后应用有冲突风险的补丁

### 校验流程

1. 使用本工具进行预校验
2. 检查 FAILED 状态的 hunks
3. 评估 VARIATION 状态的 hunks
4. 手动处理无法自动应用的补丁
5. 最终使用 `git apply` 实际应用
