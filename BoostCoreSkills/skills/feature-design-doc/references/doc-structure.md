# 设计文档结构与写作规范

这份参考定义"特性设计文档"的章节骨架与写作原则。目标读者是会去读源码、需要理解或维护该特性的工程师,因此文档要"讲清原理与设计思路",而不是堆代码或贴图。

## 黄金原则:有图、有代码,就必有文字

整篇文档最重要的一条规范:**每一张图、每一段代码,前面要有引入(这段要说明什么),后面要有解读(从图/代码里能看出什么、为什么这样设计)**。禁止出现"光秃秃的一张图""一大段代码不解释"。一段好的详细设计读起来应是"文字为主、图与代码为佐证"。

## 标准章节骨架

按下面的顺序组织(可按特性增删子节,但主干保持):

### 0. 文档导读(可选)
一段话说明文档面向谁、各章讲什么、图表用什么形式表达。

### 1. 需求分析
- **1.1 原有机制回顾**:先讲"改之前的系统长什么样"。把特性所在子系统的原生设计讲清楚(数据结构、关键路径、约束),这是后文一切设计的背景。配 1~2 张结构/路径图。
- **1.2 问题 / 痛点**:在 1.1 的背景下,点出具体场景(如大核高并发)暴露的结构性瓶颈,逐条列举并解释成因。
- **1.3 优化目标**:用表格给出可量化目标(延迟/吞吐指标、零回退要求、平台门控等)。
- **1.4 设计约束**:不破坏默认路径、与既有特性共存、构建期/运行期边界等硬约束。

### 2. 总体方案
- **2.1 核心设计思路**:用几句话讲清"这个特性到底做了什么、为什么有效"——直接对应 1.2 的瓶颈链路。再补充关键折中(为什么选这个参数/倍率)与设计纵深(与硬件/GC/编译期的耦合)。这一节要有"多几句话的介绍与综述",不能只甩一句话。
- **2.2 构建 / 开关模型**:用表格说明特性如何被触发(编译开关、运行时探测)、与其它模式如何互斥/叠加。
- **2.3 总体框架**:一张框架图(组件/泳道)。**图后必须配"框架解读——关键模块与关系"**:逐条说明图里每个关键模块是什么、模块间的边代表什么流程。
- **2.4 关键流程 / 数据流**:一张时序或数据流图。**图后配"流程与收益解读"**:这条路径相对原生改变了什么、收益从哪来。

### 3. 特性详细设计(全文重心)
按特性的子模块拆分(每个子模块一节),每节遵循固定写法:
1. **设计原理与思路**:这块要解决什么、为什么这样设计、关键取舍——文字为主。
2. **数据 / 结构在系统中如何变换**:用原理图讲清楚"内存/数据/状态从 A 变到 B 的过程"(例如 span 如何被切分→填充→回收→放大)。这是详细设计最有价值的部分,要多画几张原理图,而不是只贴代码。
3. **关键代码**:节选最能说明设计的片段(不是整文件),代码前后都要有文字。
4. **图示**:原理图 / 时序图,按"图示形式选择"(见 diagram-guide.md)决定用 SVG 还是 PlantUML。

### 4. 特性验证效果
- **4.1 验证方法与环境**:怎么测的(对照方法、绑核、轮次、统计显著性),为什么这样测才可比。
- **4.2 正确性**:功能/回归结果,并给出"这意味着什么"的结论句。
- **4.3 性能数据**:原始 benchstat / 指标表。
- **4.4 数据分析与归因**:逐项解释每个数字为什么是这样(哪项受益最大、哪项回退、根因),最后一段总结。**只贴数字不分析是不合格的。**

### 5. 使用方法与用例(Usecase)
- **5.1 用例总览**:一张用例图(PlantUML),标出角色与典型用例,图后配"用例解读"。
- **5.2~ 构建启用 / 调试观测 / 性能基准**:可复制的命令,每段命令配一句说明。
- **5.x 适用与不适用场景**:表格,给出推荐/谨慎/不适用的判断。
- **5.x 与其他特性的关系**:互斥/叠加/依赖。

### 附:关键文件清单
表格:文件路径 → 角色。便于读者按图索骥回到源码。

## 写作语气
- 中文为主、技术准确;解释"为什么"重于罗列"是什么"。
- 与源码交叉验证:文中出现的常量名、函数名、文件名,必须与代码一致。
- 不臆造数据:验证数据来自真实运行;基线选择要诚实(例如对照应在"同等其它开关"前提下,只差被测特性本身,避免把别的特性收益算到本特性头上)。
