# 三维度抓取与写作 playbook

每个维度的抓取顺序、反爬绕行、反虚构守门、写作约束。Step 3/4/5 执行前读一次。

## 通用抓取纪律

1. **先列表后详情**：先抓源的"列表/聚合页"拿到近期 N 条（标题+日期+URL），再对入选目标抓详情页。
2. **不死磕反爬（含 Anubis）+ 反爬时用"关注点扇出"来发现而非验证**：`WebFetch` 返回 "Just a moment..." / Cloudflare 挑战 / **Anubis 工作量证明挑战墙** / 登录墙 → 立刻换路，**同一 URL 不 retry 超过 2 次**。
   - **关键教训（实测漏检根因）**：当反爬源的**列表页**打不开时，你**不知道有哪些新文章**，所以"`WebSearch '<标题> site:phoronix.com>'`"这种**标题中心**写法只能*验证已知文*、**做不了发现**——会整条漏掉你还不知道标题的高价值新文（实测就这样漏了一条海光入 GCC 的友商靶心文）。
   - **正确换路顺序**：① **逐关注点定向 `site:` 发现式搜索**（见下）→ ② 源自带 **RSS**（如 `phoronix.com/rss.php`，通常不被 Cloudflare 挡）→ ③ **Wayback CDX 枚举**（见下，兼可拿可靠发布日）→ ④ 镜像/缓存。
   - **① 逐关注点扇出搜索**：对源画像该源相关的**每一条特别关注点**各跑一条定向查询并锁定当月，例如 `WebSearch "site:phoronix.com 海光 OR Hygon GCC"`、`"site:phoronix.com LoongArch"`、`"site:phoronix.com AArch64 SVE"`；**枚举返回的全部近期条目，不止取第 1 条**。列表页不可达时，**这个关注点扇出搜索就是发现机制本身**。
   - **③ Wayback CDX 兜底**（高价值反爬源）：`curl 'https://web.archive.org/cdx/search/cdx?url=<site>/news/*&from=<YYYYMMDD>&to=<YYYYMMDD>&output=json&collapse=urlkey'` 枚举窗内 `/news/` 快照，既绕 403/补搜索召回缺口，又能从首快照时间戳/归档页 byline 拿到**可靠发布日**。
   - 已知反爬清单与绕行（实测，会增长）：
     - **Phoronix / 多数商业新闻站**（Cloudflare）→ 关注点扇出 `site:phoronix.com` + RSS + Wayback CDX；**不要**预判它"总是旧文"（它对 jemalloc 5.3.1、SLUB sheaves 等都有活跃覆盖，召回不到多半是查询太泛）。
     - **X / Twitter**（登录墙）→ 搜索转述 / nitter 镜像 / 账号对应 blog·RSS；拿不到就标"未抓通"，不编造推文。
     - **sourceware patchwork / inbox、lore.kernel.org**（Anubis 挑战墙）→ 多条绕行都通，**别一拦就记 0 覆盖**：① **pipermail 静态归档**（`sourceware.org/pipermail/<list>/<YYYY-Month>/{thread,date}.html`）② **`www.mail-archive.com/<list>`** 镜像（实测无 Anubis，可读完整线程）③ **git.kernel.org 子系统树**（如 `vbabka/slab.git`）④ **LWN.net** 深度报道 ⑤ Phoronix 搜索。
3. **绝对日期必须逐条核对（HARD）**：**列表/聚合页（GitHub commit 列表、pipermail `thread.html`/`date.html` 归档索引）给的"哪条落在窗内"极不可靠**——实测会把月初条目误标成近几天。凡**入选**条目，其日期必须**直取详情页（commit 页 / PR 页 / abs 页 / 邮件单页）核对**，不信列表页/归档索引的聚合日期。核不到精确日的（如 pipermail 单封无可靠 per-message 日期）→ 如实写"日期据索引推断、未逐封核对"，并降为低置信。用 Step 1 算出的子窗绝对区间过滤；边界外 1–2 天但读者会关心的热点可破例收录并标注实际日期。
4. **维护源覆盖账本**：每个源记一行 `源 → 抓取方式 → 候选数 → 入选数 / 降级理由`，喂给源覆盖表。
5. **每源至少尝试一次**：哪怕预判没货也要试一次（一次 WebFetch 或 WebSearch），不能跳过不试。
6. **清淡日（周末/低频领域）显式策略**：当某维度严格子窗内几乎为空（典型：周末的资讯/上游），按各维度 playbook 的「清淡日 fallback」放宽，并**在该维度小节抬头标注实际放宽到的天数**；放宽后仍空的，写一句"本窗口内无强相关新增"，**不要硬凑边界/二手条目充数**。窗口外但同主题的高价值条目可作"上下文"列出，但必须显式标"窗口外上下文"、与窗内入选条目分开。

## 📄 论文 playbook

**目标**：每领域 top 3–8 篇近期强相关论文，轻量呈现。这是"每日扫一眼"，不是深度综述。

**抓取顺序**：
1. arXiv（**HTML 页优先，别用 export API**）：`export.arxiv.org/api` 限流极凶（实测连续 429/503）。默认走 HTML：`arxiv.org/list/<cat>/recent` 拿列表 → 对入选目标抓 `arxiv.org/abs/<id>`。用源画像的分类 + 关键词按命中度 + 关注点排序，控量（每领域候选 ≤ 30，入选 ≤ 8）。确需 API 时调用间隔 ≥ 3s 且仅作补充。
2. 会议：看源画像列的顶会"近 30 天有无录用名单/proceedings 发布"或"即将召开"；有则挑强相关的几篇。
3. lab/blog/OpenReview：补充。

**反虚构守门（HARD，借 paper-digest 思路）**：
- 抓到摘要 → 可写"一句话讲清 what + why"，可含论文自己声称的量化（注明"作者称"）。
- **抓不到摘要 / 只有标题（如仅有会议录用列表）** → 只写 `标题 + 链接 + 关键词命中`，**绝不编造收益数字、绝不臆测方法**，并标低置信。宁可信息少。
- **arXiv 新 ID 抽样核对**：跨月时新论文 ID（如 6 月的 `2606.xxxxx`）外观像"未来月份"易被误判为虚构——**入选的新 ID 至少抽样 1–2 个直取 `abs/<id>` 页核对**标题/作者/日期真实存在，再写进简报。

**每篇写法**（精炼，1 条目）：
```
- **<论文标题>** ｜ <arXiv id 或会议> ｜ [链接](URL)
  - <一句话：解决什么问题 + 怎么做（抓不到就只说方向）>
  - 相关性：<为什么这个领域的人值得看（对应哪条特别关注点）>
```

## 📰 资讯 playbook

**目标**：每领域 3–8 条近 1 天（薄则近 3 天）的高相关资讯，三段式呈现。

**抓取顺序**：
1. 官方/项目博客列表页 → 入选条目详情页。
2. 技术媒体：**Phoronix / 商业新闻站反爬** → 按通用纪律 2 走**关注点扇出 `site:` 发现式搜索 + RSS + Wayback CDX**（**不是**只搜已知标题）。深度 toolchain 报道别漏 **LWN.net / developers.redhat.com** 这类一手源。
3. **论坛/discourse（如 discourse.llvm.org）按"最近活跃"扫，不是只看"新主题"**：抓 `/latest` 或按关注点相关 tag（AArch64、IR-Optimizations、Clang Frontend…）过滤——很多高价值 RFC 是**老帖新活跃**（实测漏了一条 06-15 发帖、06-24 仍活跃的 NEON→SVE 再向量化 RFC）。
4. **X/推特**：x.com 直连基本拿不到。换路：
   - `WebSearch "<@handle> <关键词>"` 找被引用/转述的推文内容；
   - nitter 类镜像（常不稳定，能拿则拿）；
   - 该账号对应的 blog/RSS（很多 KOL 推文是博客的镜像）。
   - 都拿不到 → 在源覆盖表标"X 源未抓通"，**不要编造推文内容**。
5. newsletter：抓当期，挑命中关注点的条目。

**命中关注点的高价值条目优先破窗**：友商动向（海光/龙芯/Intel/AMD）、主版本发布这类正中关注点的资讯，回看窗从近 1–3 天**放宽到近 7 天**，以"窗口外·命中关注点"里程碑补充收录并标注实际日期（与其他里程碑补充同等待遇）。**未直取核对真实日期前，禁止给任一候选贴"超窗"标签**——实测就出现"非友商条放宽到 06-18/06-22 却把 06-23 海光条判超窗"的前后不一致。

**每条三段式（不可省）**：
```
### <YYYY-MM-DD> ｜ <人话标题>
- **URL**：<原文链接>
- **背景**：<1 句话，最多 2 句：为什么这件事现在出现>
- **要点**：
  - <含具体数字/版本号/CPU·ISA 名等可核验细节>
  - <...>
- **启发**：<1–2 句 actionable：在做 X 的读者下一步该 Y>
```
- 背景 1 句；要点 2–5 个 bullet（单条要点别超 10 行）；启发 1–2 句。
- 抓不到原文只有搜索摘要 → 写"基于 X 页面摘要整理"，不要伪装成读了原文。

**空资讯 / 偏科日策略**：资讯维度本就低频，清淡日（周末/niche 领域）窗内常 0 条一手报道。此时：
- 该小节写一句"本窗口内无强相关一手报道"，把窗外背景文单独列为"背景参考（非今日新发，注明日期）"，**不充作今日条目**。
- 若当天**资讯空但上游/论文有料**，把上游/论文里挖到的"故事线"（如某分配器一连串提交背后的主题、某后端的 codegen 主线）提炼进「今日脉络」，并在资讯小节末尾一句"今天真正的新动向在 💻 上游"，引导读者，别让读者以为今天没东西。

## 💻 上游代码 playbook

**目标**：每领域 3–10 条近 1 天（邮件列表近 3 天）的上游动态，按主题聚类，突出高价值条目。借 `upstream-tech-radar` 的三层深度，但做按天轻量版。

**抓取顺序**：
1. 主仓库：近期 commit（`<repo>/commits`）+ open/merged PR。优先 release notes/changelog，不要 commit-by-commit 平铺。看源画像指定的关注子目录（性能关键路径）。
   - **path 过滤别把关注点挡在外面（HARD）**：把 commit 扫描收窄到单一子目录前，先回头看本类型的**全部关注点**——若有「友商」或「跨架构」关注点，主仓的 commit 扫描就**不能只 path-filter 到一个后端目录**。典型：GCC 的友商关注点=Intel/AMD/海光/龙芯，所以 commit 必须覆盖 `gcc/config/aarch64/` **加** `gcc/config/i386/`（x86 codegen）**加** `gcc/config/loongarch/`，不能只扫 aarch64。**实测教训**：v2 GCC 只扫 `config/aarch64` 漏了 `config/i386` 的 `memset/memcpy→memmove`（b64fc29，Intel，SPEC2026 体积 −9%）——它命中友商关注点、却因 path 收窄整条没进候选池，资讯侧又恰好没有它的 Phoronix 文章兜底。**收窄路径是为省 token，但绝不能以牺牲某条关注点的覆盖为代价。**
2. **单独查 GitHub Releases/tags（与 commits 分开的一步）**：主版本/里程碑（如 jemalloc 5.3.1）只在 Releases 页可见，commit 流里看不出。**每个主仓都要查一次 releases**，源覆盖表加一列"releases 是否检查"（实测只对部分仓查了 releases、漏了 jemalloc 5.3.1 这种 4 年来首个 release）。
3. patch archive / 邮件列表：抓当期归档（pipermail/lore），按关键词过滤——**邮件列表噪声极高**，过滤后 0 条强相关就降级并量化（"归档 N 条，过滤后 0 条相关"）。lore/patchwork/inbox 常被 Anubis 拦，按通用纪律 2 多路绕行（pipermail / mail-archive.com 镜像 / git.kernel.org 子树 / LWN），**别一拦就记 0 覆盖**。
4. peer 仓库：同期看一眼，识别不同技术路线。

**上游专属约束**：
- **入窗以"合入/提交(merge/commit)日"为准**，不是作者写作日。评审滞留数月后才落 master 的补丁，按落 master 的日期入窗（写明"作者写作日 X、合入 Y"）。
- **清淡日 fallback 到近 7 天**：周末/低频领域严格近 1 天常空——放宽到近 7 天取高价值同主题条目，但把窗外的明确标为"窗口外上下文/同主题"，与窗内入选分开（别让上下文冒充今日新增）。
- **链接与 sha 纪律**：commit 用**全 40 位 sha** 的链接（短 sha 欠持久）。邮件列表**链接到归档索引页 + 引用 message 号**，不要拼造单封 message 的 URL、不要假装核过单封精确日期。

**三层深度（按价值分配笔墨）**：
- 普通条目：一句"改了什么 + 为什么重要"。
- 高价值条目（性能/向量化/架构路线）：补讨论/review/diff，提炼争议点、触达的热路径、对本领域(及 Arm·性能)的意义。
- 指令级条目（SVE/Neon/原子等）：再说明指令机制、收益来源、适用与不适用场景。

**每条写法**：
```
- **<人话标题>** ｜ <commit/PR 链接>
  - 改了什么：<...>
  - 为什么重要 / 对本领域的意义：<是 plumbing 还是真改了关键路径；对性能/Arm 的影响；短期修补还是方向信号>
```

**反虚构**：拿不到 diff/讨论就说"仅据标题+简介，深度受限"，不要假装看过代码。

## 跨维度去重（与跨天去重区分）

- **跨天**：读 `state/<标签>-seen.md` 剔除历史已报道（Step 4 主流程做）。
- **跨维度**：同一事件可能在资讯和上游都出现（如"X 发布 vN"既是新闻又是 release）。择主维度收一次，另一维度可一句话带过并链接，不要两处全文重复。
