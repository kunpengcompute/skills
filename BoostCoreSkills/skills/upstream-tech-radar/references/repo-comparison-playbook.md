# Peer Repository 对比手册

当你比较主上游仓库和一个或多个 peer repository 时，使用这份手册。默认目标不是分胜负，而是为本地 Arm 高性能开发工作提炼可用经验。

## 目标

不要把仓库对比写成“谁更强”的排行榜。真正目标是回答：

- 谁在解决什么问题
- 谁的路线更适合 Arm 平台
- 哪些做法值得本地项目迁移
- 哪些做法只是特定背景下成立

## 对比轴

优先沿这些轴对比 peer repo：

- 性能路线：吞吐、延迟、内存效率、批处理、zero-copy、cache 行为
- 向量化与架构策略：SIMD、Neon、SVE、架构特化路径
- 功能广度与聚焦程度
- API 易用性与稳定性
- 可移植性与平台策略
- 可靠性纪律：tests、fuzzing、CI、bug 处理
- 内部可维护性与贡献者友好度
- 生态姿态：bindings、integrations、packaging、documentation

## 解读 Peer 变化时要问的问题

当 peer repo 有重要变化时，至少要问：

- 它为什么要做这个变化？
- 这是领先性创新、追赶补课，还是维护债偿还？
- 这个变化是否特别照顾 Arm 或其他特定架构？
- 这个变化是否符合主上游的技术哲学？
- 这个变化是否比主上游更适合本地项目的目标？

## 产出洞察，而不是对称摘要

对比部分不需要强行平均篇幅。如果证据不均衡，可以明确写：

- 这个 peer repo 本月比较安静
- 这个 peer repo 主要在做维护性工作
- 这个 peer repo 在探索主上游尚未优先投入的方向

这种不对称本身就是结论的一部分。

## 转成本地建议

对每个有意义的 peer 变化，归到以下一类：

- 可直接迁移
- 需要适配后迁移
- 适合作为反例或风险提示
- 与本地项目当前无关

并补一句原因，尤其说明：

- 对 Arm 平台是否成立
- 是否依赖特定硬件、编译器或工程前提
- 本地项目是否具备复制该策略的条件
