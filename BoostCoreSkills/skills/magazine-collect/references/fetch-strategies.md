# 抓取策略与防爬应对

本文档解决月刊收集中**最常翻车的一步**：怎么从各类信息源把内容稳定地拿回来。重点是给出**绕行手册**，让你在遇到反爬时不要死磕同一个 URL。

**本文档与领域解耦**——讨论的是"项目官方博客 / Weekly / 代码仓 / 邮件列表 / 学术站 / 公司新闻 / 技术媒体"这类**源类型**的通用抓取策略；具体领域的关键词、源 URL、关注点都来自规则文件，不在本文档预设。

---

## 0. 覆盖完整性原则（HARD RULE）

规则文件列了 N 个源，**默认全跑**。"全跑"的意思不是"每个源都把 3 个月的内容全读完"，而是：

- **每个源至少有一次抓取尝试**（一次 WebFetch 或一次 WebSearch），不能没尝试就跳过
- **每个源都要进"源覆盖表"**（详见 `output-template.md`），写明"抓取方式 / 候选条目 / 入选条目 / 备注"
- **降级要明示**：如果你觉得某源高噪声、内容外、抓取失败，可以降级，但必须在源覆盖表里说清楚理由，**让用户能审计、能反驳、能要求你重抓**

合法降级场景与禁用降级场景：

| 场景 | 是否允许降级 |
|---|---|
| 源被反爬，镜像/搜索绕路也失败 | ✅ 标注"无法抓取" |
| 源信噪比极低（如邮件列表月新增 1 万条），抓取成本远超收益 | ✅ 但要**量化**："归档 ~1.2 万条 → 关键词过滤后 0 条强相关" |
| 内容在时间窗外（如 Weekly 期号超过窗口） | ✅ 标注覆盖期数 |
| Weekly 类站点，"读首尾两期应该差不多" | ❌ **不允许**——这是你替用户做决定。Weekly 是高密度 curated 源，13 期就有 13 期信息，必须 cover 完 |
| "我感觉它不重要" | ❌ **不允许**——重要不重要由 Step 4"特别关注点"命中判定决定，不在抓取阶段拒收 |
| 代码仓 commit 太多读不完 | ⚠️ 部分允许：优先读 release notes / changelog / discussion；只在你确认 release notes 已经覆盖了主要事件时才跳过 commit 级 |

## 总体决策流

```
┌───────────────────────────────────────────────┐
│ 拿到一个目标 URL（来自规则文件的"信息源"列表）│
└───────────┬───────────────────────────────────┘
            ↓
┌───────────────────────────────────────────────┐
│ Step A：试 WebFetch <url>                       │
└───────────┬───────────────────────────────────┘
            ↓
   ┌────────┴────────┐
   ↓                 ↓
 成功            失败/被挑战
   │             （Cloudflare、403、空内容、
   │              "Just a moment..."）
   │                 ↓
   │          ┌──────────────────────────────┐
   │          │ Step B：换镜像 / 列表页 / RSS  │
   │          └─────────┬──────────────────┘
   │                    ↓
   │             ┌──────┴───────┐
   │             ↓              ↓
   │           成功           还失败
   │             │              ↓
   │             │      ┌──────────────────────┐
   │             │      │ Step C：WebSearch 拿摘要 │
   │             │      │ + 镜像/转述源          │
   │             │      └─────────┬────────────┘
   │             │                ↓
   │             │         ┌──────┴──────┐
   │             │         ↓             ↓
   │             │       成功          仍失败
   │             │         │             ↓
   │             │         │     标记为"无法核实，跳过"
   │             │         │     在源覆盖表的备注里说明
   │             ↓         ↓             │
   └──────►汇入收集池      │             │
                           │             │
                           ↓             ↓
                    在 digest 的"源覆盖表"备注里坦白来源
```

**铁律：同一 URL 最多 retry 2 次**。第二次还失败立刻进入 Step B/C，不要陷入死循环。

## 通用源类型与推荐策略

下面按**源类型**而非具体领域分类。任何领域的规则文件，其信息源都能归到这几类的某一类。

### 1. 反爬严重型（典型如 Cloudflare 防护）

**特征**：
- 列表页和文章页都启用反爬防护
- `WebFetch` 通常返回 `"Just a moment..."` 挑战页或 403
- `curl` 同理失败
- 邮件列表/论坛镜像/搜索引擎缓存通常能访问

**判别方式**：第一次 `WebFetch` 返回非常短（< 1KB）且含"Just a moment"、"Checking your browser"、"cf-ray"字样，基本就是被 Cloudflare 拦了。

**推荐策略**：
1. **先试** `WebFetch <列表页 URL>`（有时能拿到 cached 版本）
2. 失败 → `WebSearch "site:<域名> <关键词> <月份>"` 拿搜索引擎索引的标题+摘要
3. 关键事件再用 `WebSearch "<事件标题> <相关镜像站>"` 找镜像源：
   - 同领域的转载/聚合站
   - 论坛（往往会引用文章片段）
   - `daily.dev` / `news.ycombinator.com` / `reddit.com/r/<相关 subreddit>` 讨论
   - X / Twitter 上对应作者/账号的原帖
4. 综合多个二手源后，在产物的"源覆盖表"备注列明确写：

   > 主站被反爬，本汇总基于搜索引擎索引/缓存的页面摘要 + 镜像源转述整理，文末附原文链接；摘要中如有少量数据偏差以原文为准。

### 2. 项目官方博客 / Weekly 类（金矿）

**特征**：
- 大多数是友好的，可直接 `WebFetch`
- 部分大公司站可能有地理/速率限制，但通常不阻断
- Weekly 类（如 `llvmweekly`、`this-week-in-rust`、`recsysjobs/recsys` 类 newsletter、领域内的 weekly 聚合站）是金矿——它们已经做了一遍人工 curate，**优先用它们做 cross-reference**

**推荐策略**：
1. `WebFetch` 列表页 → 拿到近 3 个月的所有 issue/post 链接
2. 对每个 post 再 `WebFetch` 详情
3. Weekly 类站点的 issue 通常是周更，3 个月约 12~13 期。**逐期读完**——不要只读首尾。每期 ~5KB，13 期总量约 65KB，远低于 token 预算；价值密度极高，跳过任何一期都可能漏掉关键事件。
4. 如果 token 真的紧张，最低降级是"读首/中/尾各两期"覆盖月份分布；但要在源覆盖表里写明"仅覆盖 6/13 期"，让用户知道。
5. 失败时同"反爬严重型"策略走 `WebSearch`

### 3. 代码仓类（结构化源）

**包括**：GitHub / GitLab / Gitee / 自托管 Git 平台上的项目主仓。

**特征**：
- `WebFetch` 友好
- 但 commit log 海量，逐 commit 读没意义、也会爆 token
- Release notes 和 changelog 才是高密度信息源

**推荐策略**（按优先级）：
1. `https://<host>/<org>/<repo>/releases`：拿近 3 个月所有 release tag，每个 release 页面有官方 release notes
2. 大版本的 `RELEASE_NOTES.md` / `CHANGELOG.md` 通常远比 commit log 有用
3. Tag 之间的 compare 页（如果你想知道两个版本之间的 commit 差）
4. **不要** 试图 `WebFetch` 整个 commits 列表后再筛——会浪费 token 且 LLM 也消化不了
5. 特殊：discussion / RFC 类 issue 是金矿（如各项目的 discourse 论坛、PEP/JEP/RFC 索引），可以用 `WebSearch "site:<discussion-host> <关键词>"` 找
6. **注意**：有些组织把镜像仓和发布仓分开（如 `gcc-mirror/gcc` 只是镜像，真实 release 在官方 mailing list 或站点上）。规则文件指向这种镜像时，要从规则文件其它字段或同领域 sources-catalog 里找出真实发布渠道。

### 4. 邮件列表（mailing list）

**包括**：各项目的 pipermail / mailman 归档。

**特征**：
- 通常 `WebFetch` 可以
- 内容噪声极大（一个月可能上万条），不可能逐条读
- 月度归档页（如 `pipermail/<list>/YYYY-Month/`）可拿目录

**推荐策略（含降级流程，规则文件列了就必须跑）**：

1. **第一步：尝试**——`WebFetch` 对应月份的归档 index 页，看返回内容和条目大致量级
2. **第二步：用关键词在归档内做强过滤**——**关键词来自规则文件该板块的"特别关注点"字段**（skill 不预设关键词；规则文件让你关注什么就过滤什么）
3. **第三步：评估强相关条目**：
   - 如果过滤后有 N 条强相关（N >= 3），**入选**这些条目，正常进事件池
   - 如果过滤后 0~2 条，**说明该月归档对当前关注点几乎没贡献**，在源覆盖表里写"归档约 X 条 → 关键词过滤后 Y 条强相关"作为合法降级
4. **第四步：找 weekly 反向定位**——若同领域有 weekly 类聚合源，看它们引用的 mailing list 主题来反向定位关键邮件

**严禁**：不试就跳过、"我感觉邮件列表都是噪声"——这是替用户做决定。每个邮件列表源至少要走完第一/二步并把结果记入源覆盖表。

### 5. 学术 / 论文类

**包括**：arXiv 列表页、各会议 proceedings 站（如 ACM DL、IEEE Xplore、各顶会官网）。

**特征**：
- arXiv 列表页 `WebFetch` 可用
- 但大类（如 cs.AR / cs.PL / cs.IR）每天就有几十篇，3 个月会有数千篇，不能全读
- ACM DL / IEEE 多数论文需要订阅才能拿全文，只能拿 abstract

**推荐策略**：
1. 优先看会议 proceedings 而不是 arXiv 海洋。按规则文件该板块涉及的领域选对应顶会（不同领域有不同的"必看会议"，由规则文件指定）
2. arXiv：用 `WebSearch "site:arxiv.org <规则文件该板块特别关注点关键词> YYYY"` 收敛到 10~20 篇候选
3. 每篇只读 abstract 即可，不要试图 `WebFetch` PDF
4. 标题/摘要里若命中规则文件该板块"特别关注点"里的具体对象/指标，直接优先

### 6. 公司新闻 / newsroom / 开发者中心

**包括**：各厂商 newsroom、developer 站、技术博客。

**特征**：
- 通常可 `WebFetch`
- 但内容市场属性强，要识别"真技术"和"PR 稿"
- 中文站有时会触发跨境访问问题，可换镜像

**推荐策略**：
1. 列表页直接 `WebFetch`
2. PR 稿（关键词："announces"、"unveils"、"leadership"、"transforms"、"率先"、"全球首发"、"重磅发布"）降权
3. 真技术（"benchmark"、"release"、"architecture"、"deep dive"、"performance"、"实测"、"实战"、"评测"后跟具体数据）保留

### 7. 技术媒体 / 综合资讯站

**包括**：InfoQ、InfoWorld、各领域的技术媒体（如分布式数据库领域的 DB-Engines blog、推荐系统领域的 RecSys community 站等）。

**特征**：
- 友好度参差不齐
- 部分付费墙
- 翻译/转载站较多，注意溯源

**推荐策略**：
1. 列表页直接 `WebFetch`，付费墙则看 abstract
2. 注意 dedup——同一事件常在多个媒体出现，引用最权威的一个 URL（通常是官方源 > 媒体原创深度报道 > 普通转载）

## 规则文件"特别关注点"的反向搜索增强（通用模式）

规则文件几乎所有板块都有"特别关注点"字段，里面通常是**读者业务直接相关的具体对象**——可能是某些硬件目标、某些竞品、某些指标、某些场景、某些算法、某些客户问题。这一类内容**单靠抓取主信息源拿不到全**——大多隐藏在 commit message 的 target 信息、release notes 的"new targets"段落、benchmark 文章的横评数据、案例 study 的客户名等位置。

**通用增强策略**（不分领域，按以下三步执行）：

1. **正向搜索**：把规则文件该板块"特别关注点"字段里的每一个具体名字/指标/对象取出来，组合 `WebSearch` query：
   - `"<关注点 A>" <板块名> <year>`
   - `"<关注点 B>" OR "<关注点 B 的别名>" <板块名>`
   - `"<对照对象>" vs "<本方对象>" <板块名>`
   - `<关注点关键词> benchmark <year>`

   **关键**：关键词从规则文件来，不要凭记忆/上次跑的领域填——例如换成搜推广领域，关键词就是规则文件里的"召回算法/排序模型/CTR/CVR/嵌入维度/MMoE/对手平台名"等。

2. **反向标注**：抓到的每条事件，**强制在内部做一遍"是否命中规则文件该板块特别关注点"的判定**。判定为"命中"才入选 Step 4 的"核心事件"段；"中性"且属"大版本/跨板块重大"才入选；"无关"直接舍弃。

3. **横评 / 对照类内容必收**：任何同领域内 "A vs B" 横评、跨厂商对比、对手 benchmark 复现类文章必收——它们是把规则文件"特别关注点"具象化的最佳载体。

## 抓取阶段的 token 节约

- WebFetch 一个长页面会消耗很多 token。如果只想要标题列表/最新几条，可在 prompt 里说明"只关注最新 20 条"
- 不要 WebFetch 同一站点超过 ~10 个详情页，超过这个数量信噪比会快速下降
- 优先用 weekly / changelog / release notes 这类"已 curate"的源
- 长论文（arXiv PDF）不要 WebFetch，只读 abstract 页

## 失败诊断速查

| 现象 | 可能原因 | 应对 |
|---|---|---|
| WebFetch 返回 "Just a moment..." | Cloudflare 反爬 | 走 Step C 的 WebSearch + 镜像 |
| 返回 403 / 401 | 站点封禁 / 需要登录 | 同上 |
| 返回内容很短 / 空 | JS 渲染页 | 试列表/RSS 版本，或走 WebSearch |
| 返回大段 HTML 但找不到正文 | 站点结构特殊 | 用 `Grep` 在返回内容里搜关键词定位 |
| 内容是英文但用户要中文 digest | 正常 | 在 Step 5 翻译/整理时统一转中文，技术术语保留英文 |
| 同一事件在多个源被报道 | 正常 | 去重，引用最权威的一个 URL（通常是官方源 > 主流媒体 > 论坛） |
| 跨境站点 reachability 不稳定 | 网络问题 | 换镜像或转用搜索引擎缓存 |

## 与 `sources-catalog.md` 的关系

本文档讲**通用源类型**的抓取手法；`sources-catalog.md` 讲**具体已沉淀的源**的友好度、建议路径、替代源。新领域跑完一轮后，把该领域的具体源经验沉淀到 `sources-catalog.md`，本文档的通用策略保持不变。
