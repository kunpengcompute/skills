## 一、本期核心脉络（一句话总结）

- **大版本**：**LLVM/Clang 22.1.0** 于 2 月 24 日发布；季度内连发 6 个双周点版本到 22.1.6，进入稳定收敛期。
- **AArch64 / 鲲鹏路线**：Arm C1 全系 + Ampere1C 在 22.1 落地；**Arm C1-Ultra 完整调度模型**、**AArch64 64 字节函数对齐 layout pass**、**C1-Nano 调度模型** 合入 LLVM 23 主干。
- **AArch64 SVE/SME codegen**：邮件列表持续上游 SVE/SME spill/fill 多向量化、`svldr_vnum_za` codegen 改进、regalloc distinct predicate 等性能补丁，多来自 Arm/Apple 工程师。
- **友商动作**：**NVIDIA Olympus**（Vera-Rubin 服务器用，Neoverse V2 衍生 + SVE2_AES/SHA3/SM4）完整调度模型合入 LLVM 22；**AMD HIP** 默认切到 LLVM 新 offload driver。
- **构建/链接性能**：**DTLTO** 单文件加链接耗时大幅下降；**MTPC** 多线程并行编译提案，模块内函数级并行，实验最高 −32% 全应用编译时间。

## 二、核心事件（按日期降序，最新在上）

### 2026-05-22 ｜ MTPC 提案：模块内函数级并行编译，最高 −32% 构建时间

- **URL**：<https://www.phoronix.com/news/LLVM-ThinLTO-MTPC-Proposal>
- **背景**：ThinLTO 现状只在 *模块级* 提供并行，单 module 内部的 codegen 仍串行，在 32~96 核服务器上未充分利用机器。
- **要点**：
  - 在 ThinLTO 框架内引入 **Multi-Thread Parallel Compilation**，按 **函数级别切分模块**并行 codegen
  - 实验数据：full application build 最多 **−32% 总时间**
  - 已有几个 PR 铺设基础（initial proposal in LLVM Discourse）
  - 与 DTLTO 形成"link 间并行 × link 内并行"互补
- **启发**：鲲鹏大型业务构建（数据库、内核、中间件 LTO build）值得提前关注；如果你团队 build 基础设施耦合到 module 切分方式，未来 1~2 个 LLVM 大版本可能要重新评估。

### 2026-05-19 ｜ LLVM 22.1.0 → 22.1.6 双周点版本节奏（含 GA 主版本）

- **URL**：<https://github.com/llvm/llvm-project/releases> ｜ <https://www.phoronix.com/news/LLVM-Clang-22.1-Released>
- **背景**：LLVM 22 系列在 2025-12 完成 feature freeze 并从主干分支，22.1.0 为 2026 首个稳定 feature 版本，并启动双周节奏点版本以快速吸收回归。
- **要点**：
  - **22.1.0** (Feb 24)：Arm **C1 Nano/Pro/Premium/Ultra** 全系基础 enablement、**Armv9.7-A (2025)** 汇编/反汇编、**Ampere1C**（推测对应 AmpereOne Aurora）；Intel **Wildcat Lake** / **Nova Lake**（APX + AVX10.2）；AMD Zen 4 优化补齐 + SPIR-V BFloat16
  - **22.1.1** (Mar 11) / **22.1.2** (Mar 24) / **22.1.3** (Apr 7) / **22.1.4** (Apr 21) / **22.1.5** (May 5) / **22.1.6** (May 19)：每个点版本主要为前两周内 P1 类回归修复 + 平台/ABI 细节
  - 同期：AVX10 256-bit 单一形态选项移除（Intel 放弃）；Intel 上游 libsycl；彻底移除 Google NaCl 残留
- **启发**：22.1.4+ 可作为发行版推下游的起点；企业内部建议每月 cherry-pick 一次，避免每次集成成本过高。鲲鹏侧若维护私有 `-mcpu`（如 tsv110 类自定义），可开始 PoC 切换主线 `-mcpu`。

### 2026-05-18 ｜ AArch64 64 字节函数对齐 layout pass + C1-Nano 调度合入主干

- **URL**：<https://llvmweekly.org/issue/646> ｜ commit <https://github.com/llvm/llvm-project/commit/899663966c7e>
- **背景**：AArch64 大核 CPU 前端取指对 64 字节缓存行边界敏感，跨缓存行指令对会产生 fetch penalty，多个 vendor（Neoverse、Apple、HiSilicon 鲲鹏代次）都建议热点函数对齐。
- **要点**：
  - 新增 **AArch64 code layout 优化 pass**：特定条件下强制 **函数 64 字节对齐**
  - 对某些 **可能跨缓存行的指令对** 也做指令对齐
  - 同周合入 **AArch64 C1-Nano 调度模型**（C1 全系调度模型陆续到位）
- **启发**：把这个 pass 纳入鲲鹏 / 大核 ARM 服务器的基线对照——通常带来 0~3% 性能浮动，正负皆有可能；如果你团队维护私有调度模型，可参考 C1-Nano 模型 patch 形态做对照。

### 2026-04-24 ｜ Arm C1-Ultra 调度模型合入 LLVM/Clang 23

- **URL**：<https://www.phoronix.com/news/Arm-C1-Ultra-Sched-LLVM-Clang>
- **背景**：LLVM 22 已落地 Arm C1 全系基础 `-mcpu`（但还在用 generic 调度模型），本次给旗舰 C1-Ultra 配上完整调度。
- **要点**：
  - 调度模型 **衍生自 Neoverse V3 调度模型**，按 Arm 官方 C1-Ultra 软件优化指南做针对性调整
  - 目标：让 Armv9.3-A + SME2 的 C1-Ultra 编出最优二进制
  - Arm 宣称数据：C1-Ultra 相比 Cortex-X925 **单线程 +25%**、实际应用平均 **+15%**
  - 将随 **LLVM 23 稳定版（约 9 月）** 发布
- **启发**：Neoverse V3 衍生的调度模型可作为类代次 ARM 服务器 SoC 的"近似起点"——在没有自家调度模型时拿来跑基线对照；标志着 Arm C1 与 Neoverse 在编译器层共享底座，未来 ARM 适配步调更同步。

### 2026-04-21 ｜ AArch64 svldr_vnum_za / svstr_vnum_za codegen 去多余 SXTW（邮件列表）

- **URL**：<https://lists.llvm.org/pipermail/all-commits/Week-of-Mon-20260420/297594.html>
- **背景**：编译 SME 的 `svldr_vnum_za` / `svstr_vnum_za` 时 LLVM 多出一条 `SXTW`，而 GCC 不出，原因是 clang 用 i64 而 llvm 中间用 i32。
- **要点**：
  - Arm 的 Jonathan Thackray 把 clang + llvm + MLIR ArmSME dialect 统一为 i64，消掉多余 SXTW
  - 同步 touch `mlir/Dialect/ArmSME/IR/ArmSMEIntrinsicOps.td`，MLIR 与 LLVM intrinsic 保持一致
- **启发**：升级 LLVM 22.1.4+ 后，SME 热点循环里的 `svldr_vnum_za` 自动少一条指令；同时也是 LLVM 与 GCC 行为对齐的范例，日后 GCC vs Clang AArch64 SME 横评这类差异都会被夷平。

### 2026-04-14 ｜ AArch64 regalloc hint：MATCH/CMP 选取与 governing 不同的 predicate 寄存器（邮件列表）

- **URL**：<https://lists.llvm.org/pipermail/all-commits/Week-of-Mon-20260413/295917.html> ｜ commit <https://github.com/llvm/llvm-project/commit/0756e585f0c3>
- **背景**：SVE 的 MATCH / CMP 等 predicate-generating 指令，在 destination predicate 与 governing predicate 同寄存器时，某些 microarch 上无法并行调度。
- **要点**：
  - Arm 的 David Sherwood 在主干加 regalloc **hint**：让 destination predicate 尽量挑一个 *与 governing predicate 不同* 的寄存器
  - hint 保守：不引入新 spill/fill（不挑 callee-save 除非已分配）
  - **默认对所有核启用**；可通过 `+disable-distinct-dst-reg-cmp-match` feature 关闭
  - 在小函数的 SVE 接口（p4~p15 为 caller-preserved）场景可能反向不利
- **启发**：鲲鹏 / 任何 SVE 代次默认享受；如果你的 vendor SoC 实测在短函数 SVE workload 上反而退化，可加 `+disable-distinct-dst-reg-cmp-match` 关闭——对 SPECCPU 这种 hot loop heavy 的 benchmark 影响可能集中在 1~3 个测试点。

### 2026-04 ｜ NVIDIA Olympus CPU 调度模型合入 LLVM 22

- **URL**：<https://www.phoronix.com/news/NVIDIA-Olympus-Sched-Model>
- **背景**：NVIDIA Olympus 是 Vera CPU（搭配 Rubin GPU）的 ARM64 大核，号称比 Grace（基于 Neoverse V2）快 2 倍，每 socket 88 核；年初已落地基础 `-mcpu`，本次补完整调度。
- **要点**：
  - **基于 Armv9.2-A**，含 SVE2_BITPERM、RNG、LS64、MEMTAG、PROFILE、FAMINMAX、FP8DOT2、LUT、**SVE2_AES / SHA3 / SM4** 等
  - 来自 NVIDIA 工程师，参考其今年公开的 Olympus 软件优化指南
  - 服务于 Vera-Rubin 88 核 CPU
- **启发**：Olympus 与未来鲲鹏代次在 ISA 维度高度重叠（Armv9.x + SVE2 + SM4/AES）；Olympus 调度模型可作为 vendor tuning 设计参考样本——尤其 SVE2_AES/SHA3/SM4 的调度对鲲鹏密码加速 workload 直接相关。

### 2026-03-28 ｜ LLVM DTLTO 单文件链入性能大幅优化

- **URL**：<https://www.phoronix.com/news/LLVM-DTLTO-Faster-Files-Link>
- **背景**：DTLTO（Distributed ThinLTO）向最终 link 添加文件这一步存在已久的瓶颈，特别是 Windows + 高核心数场景。
- **要点**：
  - Ben Dunbobbin 的补丁合入后：
    - **Windows + Ryzen 16 核**：~2799 ms → **157 ms**（≈18×）
    - **Linux**：~255 ms → ~41 ms（≈6×）
  - 对 ThinLTO 大型分布式构建（Chromium、内核 LTO build）意义显著
- **启发**：鲲鹏服务器跑 distcc 类分布式编译场景值得回归测试——Linux 端 6× 提速虽不及 Windows 18× 戏剧化，但绝对数据 ms 级也意味着集群编译 wall clock 显著下降。

### 2026-03-18 ｜ AArch64 SVE 多向量 spill/fill 上游：SVE2.1/SME2 上 N×STR_ZXI → ptrue + multi-vec（邮件列表）

- **URL**：<https://lists.llvm.org/pipermail/all-commits/Week-of-Mon-20260316/289958.html> ｜ commit <https://github.com/llvm/llvm-project/commit/73a67d6d5939>
- **背景**：SVE 寄存器组的 tuple spill/fill（STR_ZZXI / LDR_ZZZZXI 等伪指令）此前展开为 N 条独立的 STR_ZXI / LDR_ZXI，SVE2.1 与 SME2 提供 multi-vector spill/fill 原语可一条搞定。
- **要点**：
  - Apple 的 Amara Emerson 在主干提交：目标支持 **SVE2.1 或 SME2** 时，直接生成 `ptrue + multi-vec` 替代 N 条单 vec
  - x2 case 在代码大小上 neutral，但 **依赖 microarch 仍可获益**（多发射、独立 issue port、更短依赖链）
  - 新增测试 `sve-multivec-spill-fill.ll`
- **启发**：鲲鹏后续 SVE2.1 / SME2 代次自动获得，确保 `-mcpu` 字符串带 SVE2.1/SME2 feature bit；性能基线团队重点跟踪含 SVE tuple 寄存器 spill 的热点（向量化 BLAS / 卷积 kernel），可能 0~10% 改善。

### 2026-02-24 ｜ AMD HIP 默认改用 LLVM 新 offload driver

- **URL**：<https://www.phoronix.com/news/AMD-HIP-Default-New-Offload>
- **背景**：CUDA 已先于本季度切到 LLVM 统一 new offload driver，本次 HIP 也走完这条迁移路径，AMD 与上游统一。
- **要点**：
  - 改动合入 LLVM 23 主干，AMD **HIP 默认使用新 offload driver**
  - 新驱动统一不同 offload 语言接口；支持可重分发的设备端代码链接/编译、**设备端静态库**、**设备端 LTO**
  - 同时兼容 Windows 与 Linux
- **启发**：AArch64 + AMD GPU offload 团队工具链体验进一步向 CUDA 看齐；鲲鹏 + AMD GPU 组合的 AI / HPC 场景可重新评估是否抛弃 vendor-specific HIP 编译流。
