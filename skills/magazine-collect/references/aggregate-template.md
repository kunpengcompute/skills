# 月刊拼接模板（`mode=magazine` 专用）

本文档定义 **`mode=magazine`** 下 Step 8 的月刊拼接产物 md 的精确结构与写作要求。Step 8 之前必读。

**本模板与领域解耦**——所有占位符都用 `<...>` 标注，月刊的标题、一级 section、子板块嵌套全部按**当前规则文件的实际结构**动态生成；不要复用任何具体领域的预设骨架。本文末尾给出一份"编译器月刊"作为示例填充，仅供参照。

## 设计原则

1. **月刊 = 各叶子 digest 的第一二章拼接，不做新分析**。月刊产出物不引入新事件、不二次裁剪事件三段式、不写汇总分析段。
2. **跳过各叶子的"〇、源覆盖表"**——过程透明性内容合并到月刊末尾"全月覆盖速览"，避免月刊正文 50% 是表格。
3. **层级跟随规则文件结构**——一级 section 用规则文件原标题（如"一、<规则文件第一节标题>"），多叶子节再嵌一级用 **N.M 序号标题**（如"### 3.1 Java"、"### 3.2 Python"）。子板块顺序按规则文件原顺序，序号 M 从 1 开始递增。
4. **不替读者做"全月 takeaways"**——每个叶子事件的"启发"段已经够 actionable，月刊层面再写一遍是噪声。
5. **月刊标题由规则文件决定**——规则文件首行或开头通常自带领域名（如"编译技术月刊"、"推荐系统月刊"、"数据库月刊"），直接采用；规则文件没写就用 `<领域>月刊`，领域名从规则文件主题概括。
6. **digest 链接走二级目录**：月刊正文引用各叶子 digest 时用相对路径 `./<board-slug>/<board-slug>-<start>_<end>.md`，因为月刊在 `magazine-collect-output/<domain>-<start>_<end>/` 根而 digest 在 `magazine-collect-output/<domain>-<start>_<end>/<board-slug>/`。

## Step 8 工程要点

### 8.1 抽取时的字符串边界

每份独立 digest 都符合 `output-template.md` 的精确结构。抽取脚本（实际是 LLM 心智模型）按以下规则：

- **核心脉络段**：从 `## 一、本期核心脉络（一句话总结）` 到下一个 `## ` 标题之间的所有内容
- **核心事件段**：从 `## 二、核心事件（按日期降序，最新在上）` 到文件结束或下一个 `## ` 标题之间的所有内容

### 8.2 标题降级映射

| 独立 digest 里的标题 | 月刊里的标题 |
|---|---|
| `## 一、本期核心脉络（一句话总结）` | `### 核心脉络`（单叶子节）/ `#### 核心脉络`（多叶子节子板块下） |
| `## 二、核心事件（按日期降序，最新在上）` | `### 核心事件`（单叶子节）/ `#### 核心事件`（多叶子节子板块下） |
| `### YYYY-MM-DD ｜ 标题`（事件标题） | `#### YYYY-MM-DD ｜ 标题`（单叶子节）/ `##### YYYY-MM-DD ｜ 标题`（多叶子节） |

**多叶子节子板块标题**：`### <N>.<M> <子板块名>`，其中 N 是该多叶子节的章节号（如三节就是 3），M 从 1 开始按规则文件原顺序递增。例：`### 3.1 Java` / `### 3.2 Python` / `### 4.1 异构计算` / `### 4.2 AI 编译`。

**降级一致性**：同一节内所有事件的标题级别必须一致；不要让某事件在 `####`、相邻事件却在 `#####`。

### 8.3 事件正文原样保留

事件三段（以及规则文件扩展的额外段，如某些领域规则文件可能在"总结和表达方式"里指定附加"工业界价值 / 学术界价值"或"风险 / 落地路径"等段）**原样复制**，不做：

- 不改写"背景"段（不要"为月刊读者再压缩一句"）
- 不改写"要点"段（不要去掉具体数字）
- 不删 URL、不删任何 bullet
- 不裁剪规则文件指定的扩展段——它们在叶子 digest 里就是这么写的，月刊里就这么呈现

### 8.4 一级 section 标题

直接用规则文件第一行级章节的原标题，**不重新起名**。仅在中文文字前后补空格便于阅读，不改名。

例：

| 规则文件原标题 | 月刊一级 section 标题 |
|---|---|
| `## 一、<节名 A>` | `## 一、<节名 A>` |
| `## 二、<节名 B>` | `## 二、<节名 B>` |
| `## 三、<节名 C1>/<节名 C2>/...` | `## 三、<节名 C1> / <节名 C2> / ...`（仅补空格） |

### 8.5 全月覆盖速览的合计行

末尾合计行 `| **合计** | **候选总数 → 入选总数** | **跑通源总数 / 规则列源总数** | — |` **必须**给出，让读者一眼看到全月数据量级。

## 反例

### 反例 1：把各叶子的"〇、源覆盖表"也拼进月刊正文

**不要**：把每个叶子的源覆盖表都贴进月刊里——读者会被巨型表格淹没，且各板块表头格式略有差异看着杂乱。

**要**：月刊末尾用一张"全月覆盖速览"合并表搞定，单板块详细源覆盖留在叶子独立 digest。

### 反例 2：在月刊里加"全月 takeaways / 总编辑视角"段

**不要**：

```markdown
## 全月编辑视角

回顾本月，整个 <领域> 生态呈现以下三大趋势：
1. ...
```

**要**：每个叶子事件的"启发"段已经够用；月刊层面再写一遍是冗余。如果用户主动要求"全月编辑视角"，那是另一个需求，不属于本 skill 默认行为。

### 反例 3：拼接时现场补抓事件

**不要**：发现某板块只有 4 条事件感觉太少，在拼接时跑 `WebSearch` 现场补 2 条进月刊。

**要**：要补就回头改对应叶子 digest，重跑 Step 1~7 把该叶子重新生成，然后再走 Step 8 重新拼接。月刊和叶子 digest 必须是严格的"复制 + 章节重组"关系，不能漂移。

### 反例 4：标题层级不一致

**不要**：同一节里有的事件 `####`、有的 `#####`——会显示为 markdown 渲染错乱。

**要**：按 8.2 表格里的映射，单叶子节统一 `####`、多叶子子板块下统一 `#####`。

### 反例 5：删事件细节"为了月刊紧凑"

**不要**：把独立 digest 里"要点"段的 5 个 bullet 压缩成 2 个、把 URL 删掉、把"启发"裁成 1 句。

**要**：原样保留。月刊紧凑的来源是"只拼一二章"，不是"裁剪事件细节"。

### 反例 6：套用上一次的月刊骨架（最常见误区）

**不要**：上次跑了编译器月刊得到"一、LLVM / 二、GCC / 三、Rust/Go/Java/Python/WASM..."，这次跑搜推广（或数据库、安全、Web3 等任意新领域）时还沿用同一套一级 section。

**要**：月刊一级 section 与子板块嵌套**完全按当前规则文件**展开。新领域的规则文件可能是"一、召回 / 二、排序 / 三、重排 / 四、广告策略"或"一、SQL 引擎 / 二、存储 / 三、分布式事务"——一切以当前规则文件为准。

### 反例 7：把月刊标题硬编码为某个领域

**不要**：标题写死 `# 编译技术月刊` 或 `# 推荐系统月刊` 然后跑别的领域时忘了改。

**要**：月刊标题从规则文件主题动态生成（`# <领域>月刊 <YYYY-MM>`），领域名取自规则文件首段标题或主题概括。

## 与单板块模板（`output-template.md`）的关系

| 维度 | 单板块 digest（`output-template.md`） | 月刊（本文档） |
|---|---|---|
| 触发模式 | `mode=single`（默认） | `mode=magazine` |
| 文件命名 | `<board-slug>-<start>_<end>.md` | `magazine-<start>_<end>.md` |
| 输出路径 | `magazine-collect-output/<domain_slug>-<start>_<end>/<board-slug>/<board-slug>-<start>_<end>.md` | `magazine-collect-output/<domain_slug>-<start>_<end>/magazine-<start>_<end>.md`（领域-窗口根下，与各板块子目录平级） |
| 章节 | 〇 源覆盖表 / 一 核心脉络 / 二 核心事件 | 目录 / 各板块（核心脉络 + 核心事件）/ 全月覆盖速览 |
| 事件三段 | 必须写 | 原样复制，不改 |
| 源覆盖表 | 每板块一张完整表 | 末尾一张精简合计表 |
| 何时跑 | 单板块按需 | 全板块跑完 single 后再走 Step 8 拼接（已有同期 single digest 可直接复用，不重抓） |

**核心约束**：月刊永远是单板块 digest 的"重组视图"，不引入新内容。这是月刊保持可信、可复核、可重生成的根本。
