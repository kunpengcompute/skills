# 报告模板 (Report Template)

## 概述

本文档提供路径穿越漏洞扫描报告的标准模板。核心原则：**明确区分真问题和非问题**，基于"谁控制路径"和"是否跨越信任边界"两个核心判断标准。

---

## 核心判断标准

### 两个关键问题

判断一个文件操作点是否为漏洞，只需回答两个问题：

1. **谁控制路径？** — 攻击者、管理员、开发者、内部代码？
2. **是否跨越信任边界？** — 数据是否从不可信域流入可信域？

### 判断矩阵

| 路径控制者 | 是否跨越信任边界 | 判定 | 示例 |
|-----------|-----------------|------|------|
| 攻击者 | ✅ 是 | ⚠️ 真问题 | HTTP 参数、网络报文、JNI 传入数据 |
| 攻击者（受限） | ✅ 是 | ⚠️ 真问题（需评估影响） | 环境变量（SUID 场景） |
| 管理员 | ❌ 否 | ✅ 非问题 | 配置文件字段、部署环境变量 |
| 开发者 | ❌ 否 | ✅ 非问题 | 硬编码路径、API 函数参数 |
| 内部代码 | ❌ 否 | ✅ 非问题 | 整数格式化的 /proc 路径 |

### 常见非问题场景（不要误报）

| 场景 | 为什么不是漏洞 |
|------|---------------|
| 命令行工具，用户指定路径 | 用户控制自己的程序，不是攻击 |
| 库函数 API，调用者传路径 | 调用者和被调函数在同一信任域 |
| 配置文件字段（需 root 修改） | root 是可信的，root 想做任何事直接做就行 |
| 环境变量（非 SUID 进程） | 部署系统/管理员控制，不是攻击者 |
| dlopen API 参数 | 调用者决定加载什么库，正常行为 |
| 硬编码字符串 | 无外部输入参与 |

## 真问题详情模板

### 模板 A：外部输入直接到达文件操作（高风险）

```markdown
### ⚠️ 问题 #N：[标题] — 外部输入无验证

**风险等级**: High / Critical  
**CWE**: CWE-22 (Path Traversal)  
**影响范围**: [描述]

**问题代码**:

```language
// source → sink 的关键代码路径
```

**为什么是真问题**：

1. **数据跨越了信任边界**：[具体描述，如"HTTP 请求参数直接传入文件操作"]
2. **无任何验证**：[列出缺失的验证措施]
3. **攻击者可完全控制路径**：[描述攻击者如何构造恶意路径]

**数据流**：

```
[Source] HTTP 参数 / 网络报文 / JNI 数据  [EXTERNAL]
    │
    ▼
[边界] 网络边界 / JNI 边界 / IPC 边界
    │
    ▼
[Var] 变量名 = 外部数据  [TAINTED - 未净化]
    │
    ▼
[Sink] fopen / open / mkdir / dlopen  [VULNERABLE - 无验证]
```

**攻击向量**：

```language
// 攻击示例
path = "../../etc/passwd"
```

**影响**：[具体影响描述]

**修复建议**：

```language
// 1. 检查路径穿越序列
if (path.find("..") != std::string::npos) {
    throw Exception("Invalid path");
}

// 2. 规范化路径
char *resolved = realpath(path.c_str(), nullptr);

// 3. 白名单验证
if (!isInAllowedDir(resolved)) {
    throw Exception("Path not in allowed directories");
}
```
```

### 模板 B：环境变量在 SUID 场景下的风险（中风险）

```markdown
### ⚠️ 问题 #N：[标题] — getenv() 在 SUID 场景下不安全

**风险等级**: Medium  
**CWE**: CWE-73 (External Control of File Name or Path)  
**影响范围**: [受影响的子组件]

**问题代码**:

```language
static char *xxx_getenv(const char *name)
{
    return getenv(name);  // ← SUID 场景下不安全
}
```

**为什么是真问题**：

1. **SUID 进程中 getenv() 返回用户控制的值**：攻击者可设置环境变量
2. **信任边界被跨越**：用户环境变量 → SUID 进程（提权上下文）
3. **secure_getenv() 是正确做法**：SUID/SGID 进程中返回 NULL

**对比**：

```language
// 不安全：getenv() — SUID 场景下返回用户值
return getenv(name);

// 安全：secure_getenv() — SUID/SGID 场景下返回 NULL
return secure_getenv(name);
```

**影响**：[具体影响，注意评估实际危害程度]

**修复建议**：

```language
static char *xxx_getenv(const char *name)
{
    return secure_getenv(name);  // SUID/SGID 场景下返回 NULL
}
```
```

### 模板 C：跨进程/跨语言信任边界（中风险）

```markdown
### ⚠️ 问题 #N：[标题] — 跨 [JNI/IPC/RPC] 信任边界无验证

**风险等级**: Medium  
**CWE**: CWE-22 / CWE-73  
**影响范围**: [描述]

**问题代码**:

```language
// 接收端代码 — 无验证直接使用
auto path = deserialize(data).at("pathField").get<std::string>();
mkdir(path.c_str(), 0750);  // ← 无验证
```

**为什么是真问题**：

1. **数据跨越了 [JNI/IPC/RPC] 信任边界**：[描述边界]
2. **接收端无任何验证**：[列出缺失的验证]
3. **发送端可能被攻击者控制**：[描述多租户/用户输入场景]

**数据流**：

```
[发送端] 语言A / 进程A
    │
    ▼
[边界] JNI / IPC / RPC / 共享内存
    │
    ▼
[接收端] 语言B / 进程B — 无验证直接使用
    │
    ▼
[Sink] 文件操作  [VULNERABLE]
```

**修复建议**：

```language
// 在信任边界接收端添加验证
auto path = deserialize(data).at("pathField").get<std::string>();

// 1. 检查路径穿越
if (path.find("..") != std::string::npos) {
    throw Exception("Path traversal detected");
}

// 2. 规范化并验证
char *resolved = realpath(path.c_str(), nullptr);
if (resolved) {
    std::string normalized(resolved);
    free(resolved);
    // 3. 白名单验证
    if (!isInAllowedDir(normalized)) {
        throw Exception("Path not in allowed directories");
    }
}
```
```

## 非问题详情模板

### 模板 D：配置文件控制路径（管理员信任域）

```markdown
### ✅ 非问题 #N：[标题] — 配置文件字段

**位置**: [file.c:123](file:///path/to/file.c#L123)

```language
// 从配置文件读取路径
config_lookup_string(&cfg, "log_file", &value);
fopen(value, "a");
```

**为什么不是问题**：

1. **配置文件需要管理员权限修改**：[如 /etc/app/app.conf 需要 root]
2. **管理员是可信的**：如果管理员想往某处写文件，直接写就行了，不需要绕道应用
3. **攻击前提不成立**：攻击者无法修改配置文件

> 关键判断：**谁控制路径的值？** 是管理员，不是攻击者。管理员配置路径是正常运维行为。
```

### 模板 E：API 函数参数（调用者信任域）

```markdown
### ✅ 非问题 #N：[标题] — API 函数参数

**位置**: [file.c:456](file:///path/to/file.c#L456)

```language
void load_library(const char *lib_path) {
    dlopen(lib_path, RTLD_NOW);  // ← 调用者传入路径
}
```

**为什么不是问题**：

1. **路径由 API 调用者控制**：调用者和被调函数在同一信任域
2. **这是正常的 API 行为**：库函数无法预知调用者会传什么路径，验证是调用者的责任
3. **不存在信任边界跨越**：数据在同一个进程/模块内部流动

> 关键判断：**谁控制路径的值？** 是调用者（开发者），不是攻击者。库函数接受调用者指定的路径是正常行为。
```

### 模板 F：环境变量（非 SUID 场景）

```markdown
### ✅ 非问题 #N：[标题] — 环境变量（部署系统管理）

**位置**: [file.cpp:78](file:///path/to/file.cpp#L78)

```language
auto home = std::getenv("APP_HOME");
auto realPath = realpath(home, nullptr);  // ✅ 有 realpath 防护
ifstream conf(realPath);
```

**为什么不是问题**：

1. **环境变量由部署系统/管理员设置**：不是终端用户输入
2. **应用不运行在 SUID 上下文中**：不存在提权风险
3. **代码已使用 realpath() 规范化路径**：防护措施到位
4. **在容器化部署中**：环境变量由容器编排系统控制
```

### 模板 G：硬编码路径 / 内部可控数据

```markdown
### ✅ 非问题 #N：[标题] — 硬编码路径 / 内部可控

**位置**: [file.c:89](file:///path/to/file.c#L89)

```language
snprintf(path, sizeof(path), "/proc/%d/status", tid);  // tid 是整数
fopen(path, "r");
```

**为什么不是问题**：

1. **路径格式固定**：`/proc/<整数>/status`，不存在路径操控空间
2. **整数无法注入路径穿越序列**：`%d` 格式化 `pid_t`，无法插入 `..` 或 `/`
3. **数据来源完全内部可控**：tid 来自 `readdir()` 结果
```

## 报告编写注意事项

### ⚠️ 必须遵守

1. **每个条目必须明确标注**：⚠️ 真问题 或 ✅ 非问题
2. **每个条目必须说明原因**："为什么是真问题" 或 "为什么不是问题"
3. **必须包含核心判断逻辑表**：路径控制者 + 是否跨越信任边界 → 判定
4. **非问题也要列出**：让读者知道哪些地方已检查过、为什么不是问题
5. **修复建议仅针对真问题**：非问题可提供纵深防御建议，但标注为"非漏洞"

### ❌ 常见错误

1. **把管理员控制的配置路径报为漏洞** — 管理员是可信的
2. **把 API 函数参数报为漏洞** — 调用者和被调函数在同一信任域
3. **把命令行工具的用户参数报为漏洞** — 用户控制自己的程序
4. **只列出问题，不列出非问题** — 读者无法区分哪些已检查过
5. **模糊分类** — 用"潜在风险"等模糊词汇，不敢明确判定

### ✅ 正确做法

1. **基于信任边界严格判定** — 跨越了就是真问题，没跨越就是非问题
2. **明确写出判定理由** — "谁控制路径？管理员。管理员是可信的。"
3. **非问题也要详细说明** — "为什么不是问题：1... 2... 3..."
4. **用表格汇总** — 让读者一眼看清所有判定
5. **区分 CWE-22 和 CWE-73** — 路径穿越(CWE-22) vs 外部控制路径名(CWE-73)
