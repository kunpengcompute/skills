# 持续优化闭环

## 目标

把每次适配中的人工微调和验证结果，转成下次任务可复用的规则，逐步降低人工参与度。

## 记录内容

每次适配至少记录：

- 日期
- 任务主题
- 目标版本
- 参考 patch 来源
- 候选 patch 列表
- 选择理由
- 自动 apply 结果
- `.rej` 文件数量
- 主要人工修复点
- 编译是否成功
- 安装是否成功
- 启动是否成功
- 功能验证是否成功
- 采用了哪些验证方式
- 是否提交远程
- 本次新发现规则

## 指标

建议长期跟踪以下指标：

- 参考 patch 选择命中率
  定义：首次选择的主参考 patch 最终未被替换的比例
- 自动 apply 成功率
  定义：自动应用成功文件数 / patch 修改文件总数
- 编译一次通过率
  定义：第一次完整编译就通过的任务比例
- 启动通过率
  定义：完成编译后能成功启动实例的比例
- 功能验证通过率
  定义：核心 SQL/MTR/sysbench 验证通过的比例
- 人工修复密度
  定义：人工修复点数量 / patch 修改文件数

## 更新节奏

### 每次任务后

- 向 `adaptation-cases.jsonl` 追加一条记录
- 更新最贴近的 `case-*.md` 案例

### 每 5 次同类任务后

- 汇总高频冲突点
- 把重复出现的修复方式提升到 `SKILL.md` 或 `checklists.md`
- 更新参考 patch 选择规则

### 每月

- 统计各 patch 家族的通过率
- 统计哪个参考分支最稳定
- 调整主推荐参考顺序

## 规则升级标准

满足以下任一条件，就应升级为规则库内容：

- 同类冲突连续出现 2 次以上
- 同类 patch 的最佳参考来源已经明显稳定
- 某个验证动作多次发现问题
- 某个提示词约束明显降低返工

## 输出要求

当用户要求“总结经验”或“固化 skill”时，优先输出：

1. 本次新增规则
2. 本次不再推荐的旧做法
3. 后续默认自动执行的验证动作
4. 哪些点仍保留人工决策
