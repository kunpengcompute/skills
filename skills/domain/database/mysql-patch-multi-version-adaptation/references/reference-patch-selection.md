# 参考 patch 选择规则

## 目标

在真正移植前，先从多个版本分支中选出“最值得移植的参考 patch”，减少低质量起点带来的返工。

## 必做动作

1. 列出目标仓库相关分支
2. 列出各分支 `boostdb-patches/` 中与目标能力相关的 patch
3. 对候选 patch 做打分
4. 在进度输出中解释选择理由

默认查看对象：

- `boostkit/mysql` 仓库
- 各版本分支下的 `boostdb-patches/`

## 候选打分维度

每个候选 patch 按下面五项打分，满分 10 分：

### 1. 功能一致性

- 10 分：同一能力、同一实现路线
- 7 分：同一能力，但已有部分重构
- 3 分：只是相近优化，不是同一 patch 家族
- 0 分：不具备目标能力

### 2. 版本邻近度

- 10 分：与目标版本差距 <= 1 个小版本段
- 8 分：差距 2-3 个小版本段
- 4 分：差距较大，但仍在同主线
- 0 分：跨度过大，且中间经历大量架构变化

### 3. 产品线邻近度

- 10 分：同一产品线
- 7 分：不同产品线，但共享大部分内核结构
- 3 分：分叉差异大

### 4. 验证完整度

- 10 分：含 MTR、sys_vars、debug case、性能或回归用例
- 6 分：只含部分功能测试
- 2 分：几乎没有验证材料

### 5. 历史通过率

- 10 分：历史多次适配成功，人工修复少
- 6 分：有成功记录，但修复较多
- 2 分：成功记录少或失败多

## 选择规则

### 规则 1

如果某分支没有目标能力 patch，直接排除。

例：

- 目标是 `8.0.36` 的 plan cache
- `MySQL-8.0.25` 分支没有 plan cache patch
- 因此不能把 `8.0.25` 作为主参考

### 规则 2

如果官方目标版本缺少目标 patch，而 Percona 近版本存在同能力 patch，优先参考近版本 Percona，而不是更老的官方分支。

例：

- 目标是官方 `8.0.36`
- 近版本官方分支没有成熟 plan cache patch
- `MySQL-Percona-Server-8.0.43-34` 有完整 patch 和验证
- 则主参考优先选 `Percona-8.0.43-34`

### 规则 3

如果两个候选 patch 功能都接近，优先选测试更多、修 bug 更多的那个。

### 规则 4

如果高版本 patch 改动更大，但明显包含后续 bugfix，通常应以高版本 patch 为主参考，再用低版本 patch 辅助识别回移成本。

### 规则 5

允许“双参考”策略：

- 主参考 patch：决定核心实现路线
- 辅助参考 patch：用于观察目标版本附近的命名、目录、测试基线

## 建议输出格式

```text
候选参考 patch：
1. branchA/patchA
2. branchB/patchB

排除原因：
- branchX：无同能力 patch
- branchY：版本跨度过大，且验证覆盖不足

最终主参考：
- branchA/patchA

辅助参考：
- branchB/patchB
```

## 已知实例

### plan cache 移植到 8.0.36

更优主参考：

- `MySQL-Percona-Server-8.0.43-34/boostdb-patches/0001-add-plan-cache-to-improve-select-performance.patch`

不宜作为主参考：

- `MySQL-8.0.25`

原因：

- `8.0.25` 无同能力 patch
- `8.0.43-34` 与 `8.0.36` 更接近
- `8.0.43-34` 自带更多验证和修复历史
