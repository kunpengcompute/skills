# 用例生成校验清单（自检阶段参考）

> 用途：用例生成后的自检阶段参考文档，提供校验规则、排列顺序、强制约束清单。

---

## 一、完整自检清单

### 1.1 格式检查（必须通过）

每条用例生成后必须逐条通过以下校验：

```
FOR EACH 用例:
  CHECK 字段数量 == 4
  CHECK 用例_编号 MATCHES "Spark_OmniRuntime_[A-Z][a-zA-Z]+_\d{3}"
  CHECK 用例_名称 STARTS_WITH "{算子名称}_"
  CHECK 用例_级别 == "Level1"
  CHECK 用例_测试步骤 IS object
  CHECK 用例_测试步骤 HAS step1
  FOR EACH step IN 用例_测试步骤:
    CHECK step HAS description
    CHECK step HAS sql_statement
    CHECK step HAS expected_result
  END FOR
END FOR
```

### 1.2 异常用例格式检查（必须通过）

```
FOR EACH 用例:
  IF 用例_名称 CONTAINS "异常场景" THEN
    // 异常用例只有step1，没有step2
    CHECK 用例_测试步骤 NOT HAS step2

    // 异常用例的预期结果必须是"执行报错"
    CHECK step1.expected_result CONTAINS "执行报错"

    // 异常用例的SQL必须能触发错误
    CHECK step1.sql_statement 能触发以下错误之一:
      - 列数不匹配: INSERT语句的VALUES数量与表定义不一致
      - 列名不匹配: INSERT语句指定的列名不存在
      - 类型不匹配: INSERT语句的值类型与列类型不兼容
      - CHAR超长: INSERT语句的字符串长度超过CHAR定义
      - DECIMAL溢出: INSERT语句的数值超出DECIMAL精度
      - 动态分区未开启: 未设置动态分区参数
      - 大量NULL值: 全NULL值写入（特殊场景）
  END IF
END FOR
```

### 1.3 内容检查（建议通过）

| 检查项 | 检查规则 | 严重程度 |
|:---|:---|:---|
| 名称-实现一致性 | 去掉类型/模式后，步骤SQL能区分不同用例 | ⚠️ 建议 |
| 数据类型覆盖 | 名称中标注的类型在SQL中使用了边界值 | ⚠️ 建议 |
| 功能模式体现 | 步骤SQL体现该模式的本质特征 | ⚠️ 建议 |
| 预期结果差异 | 同类用例的预期结果有差异（非完全相同） | ⚠️ 建议 |

### 1.4 数据质量检查（建议通过）

| 检查项 | 检查规则 | 严重程度 |
|:---|:---|:---|
| INSERT行数 | 非异常用例的INSERT语句包含≥20行数据 | ⚠️ 建议 |
| 边界值覆盖 | 边界值用例包含最大值/最小值/零值 | ⚠️ 建议 |
| NULL值覆盖 | 数据类型用例包含NULL值 | ⚠️ 建议 |
| 存储格式 | 写入类用例统一使用STORED AS ORC | ⚠️ 建议 |
| ORDER BY | SELECT验证语句包含ORDER BY id | ⚠️ 建议 |

## 二、用例排列顺序（三层结构）

所有算子类型的用例排列遵循统一的**三层结构**：

**第一层：基础验证层（所有算子通用，优先级最高）**

| 序号 | 用例类别 | 说明 |
| :- | :--- | :--- |
| 1 | 开关测试 | 开关开启/关闭对比，验证Omni算子触发/回退 |
| 2 | 单一数据类型基本操作 | 每种支持的数据类型各1条基本功能用例 |
| 3 | 组合类型操作 | 所有支持类型组合的1条综合用例 |
| 4 | 不支持数据类型验证 | 所有不支持类型组合的1条回退验证用例 |

**第二层：功能深化层（按算子功能语义差异化，见速查卡）**

各算子类型的功能深化层子项已在 ref-operator-scenarios.md 各速查卡中标注，只选对应算子类型的子项，不混入其他类型。

**第三层：质量保障层（所有算子通用）**

| 序号 | 用例类别 | 适用算子 | 说明 |
| :- | :--- | :--- | :--- |
| 6 | 边界值测试 | 全部算子 | 各数据类型边界值操作 |
| 7 | NULL值处理测试 | 全部算子 | 含NULL值操作（如未在前面覆盖） |
| 8 | 存储格式验证 | 全部算子 | ORC格式验证 |
| 9 | 空表/极端数据量测试 | 读取类算子 | 空表查询、单行查询、大结果集 |
| 10 | **异常测试用例** | 全部算子 | **必须放在JSON文件最末尾** |

**排列原则**：
1. 第一层必在最前：开关→单类型→组合类型→不支持类型，顺序固定
2. 第二层按算子类型选择对应速查卡中的子项
3. 第三层必在最后：边界值→NULL→格式→极端数据量→异常，异常用例永远在末尾
4. 同一类别内按Level0→Level1→Level2优先级排列

## 三、AI生成强制约束清单

1. 必须先定位算子的双维度分类（数据流向 × 功能语义），再匹配测试模式
2. 必须基于速查卡中的输入因子表生成用例，不得遗漏任何因子与取值
3. 必须覆盖速查卡中测试场景清单的所有场景，每个场景至少1条用例
4. 必须按SQL测试模式规范生成SQL语句，写入类和读取类模式不可混用
5. 必须按数据类型规范生成测试数据，INSERT数据≥20行，覆盖边界值、NULL、特殊字符
6. 必须按预期结果规范填写预期，支持类型和不支持类型的预期不可混用
7. 必须按JSON输出规范格式化，字段名称、值格式严格匹配，`用例_测试步骤`必须为嵌套对象格式
8. 必须按用例排列顺序整理，异常测试用例放在JSON文件最末尾
9. 所有用例名称必须以`{算子名称}_`开头
10. 每条用例必须明确标注覆盖的准则编号、功能点编号或分支编号，确保可追溯
