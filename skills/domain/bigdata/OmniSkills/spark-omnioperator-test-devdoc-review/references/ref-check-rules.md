# 设计文档检视规则与判定标准

---

## 核心检视原则

| # | 原则 | 说明 |
|:---|:---|:---|
| 1 | 功能性部分是强制检视项 | F1~F9 任意不通过 → 最终结论"不通过" |
| 2 | 非功能性缺失不影响结论 | 非功能性章节缺失 → 仅给修改建议 |
| 3 | 不凭空补充 | 不编造文档中没有的信息 |
| 4 | 未说明即标记 | 文档未说明 → 明确标记为"未说明" |
| 5 | 结构化输出 | 便于需求、开发、测试人员直接修改文档 |
| 6 | 面向测试生成 | 每个检视项都应回答：agent 能据此生成测试用例吗？ |

## 判定等级

| 判定 | 符号 | 含义 | 对最终结论的影响 |
|:---|:---|:---|:---|
| 通过 | ✅ | 检视项符合要求 | 不影响 |
| 不通过 | ❌ | 检视项不符合要求 | 功能性不通过 → 最终"不通过" |
| 建议 | 💡 | 非功能性的改进建议 | 不影响最终结论 |

## F1: 测试约束

| 检视项 | 通过条件 | 不通过条件 |
|:---|:---|:---|
| 测试约束是否说明 | 明确说明测试约束，内容具体 | 未提及，或仅模糊描述 |

**文档中应至少包含以下测试约束要素**：
- 测试方法（如双路径对比测试法）
- 变量控制策略
- 对标基准
- 环境要求
- 数据规范

**F1 判定**：约束内容具体且可执行 → ✅；缺失或模糊 → ❌

**F2 判定**：19 种全部明确 → ✅；任意缺失或模糊 → ❌

## F3: 开关

| 检视项 | 通过条件 | 不通过条件 |
|:---|:---|:---|
| 是否有开关 | 明确说明"有开关"或"无开关" | 未提及 |
| 开关名称 | 提供具体配置项名称 | 未提供 |
| 默认值 | 说明默认值 | 未说明 |
| 开启方式 | 说明如何开启 | 未说明 |
| 关闭方式 | 说明如何关闭 | 未说明 |
| 生效范围 | 说明生效范围 | 未说明 |
| 影响范围 | 说明影响范围 | 未说明 |

**特殊规则**：文档明确"无开关"时，开关名称/默认值/开启/关闭/生效范围/影响范围不检视，该项自动通过。

**F3 判定**：所有适用项通过 → ✅；任意适用项不通过 → ❌

## F4: 观测算子

| 检视项 | 通过条件 | 不通过条件 |
|:---|:---|:---|
| 算子名称 | 具体的 Omni 算子名称 | 未提供或模糊 |
| 触发条件 | 清楚描述何时触发 | 未说明 |
| 如何判断生效 | 验证算子激活的方法 | 未说明 |
| 如何判断回退 | 验证算子回退的方法 | 未说明 |
| 如何判断走 Omni | 验证 Omni 路径执行的方法 | 未说明 |
| 观察方式 | 具体方法（EXPLAIN plan、日志关键字、UI、指标等） | 未说明 |

**F4 判定**：6 项全部通过 → ✅；任意不通过 → ❌

## F5: 设计流程

| 检视项 | 通过条件 | 不通过条件 |
|:---|:---|:---|
| 主流程 | 主执行流程清晰描述 | 未描述 |
| 异常分支 | 异常/错误分支有描述 | 未描述 |
| 流程图 | 提供流程图、时序图或清晰的步骤说明 | 未提供 |

**F5 判定**：3 项全部通过 → ✅；任意不通过 → ❌

## F6: SQL 用例示例

每条 SQL 用例必须包含以下信息：

| 检视项 | 通过条件 | 不通过条件 |
|:---|:---|:---|
| SQL 样例 | 至少一个可执行 SQL 示例 | 无 SQL 示例 |
| 测试目的 | 说明用例的测试目的 | 未说明 |
| 前置条件 | 说明执行前置条件 | 未说明 |
| 输入数据 | 说明测试数据 | 未提供 |
| SQL 语句 | 完整 SQL 语句 | 不完整或仅占位符 |
| 预期结果 | 说明预期结果 | 未说明 |
| 观察点 | 说明需要观察的内容 | 未说明 |
| 是否预期走 Omni | 明确说明 | 未说明 |
| 是否预期回退 | 明确说明 | 未说明 |

**F6 判定**：所有用例的所有 9 项均通过 → ✅；任意用例任意项不通过 → ❌

**F7 判定**：5 个维度全部覆盖 → ✅；任意缺失 → ❌

## F8: 行为可测试性（NEW）

> 此维度评估文档是否提供了足够的行为细节，使下游 agent 能够据此生成具体、可执行的测试用例。
>
> **核心逻辑**：测试设计 agent 不了解算子的内部逻辑，它完全依赖设计文档中的行为描述来生成测试用例。如果文档只说"不支持 ARRAY 类型"而不说明"遇到 ARRAY 类型时会怎样"，agent 就无法生成对应的异常/回退测试用例。

### 检视项

| # | 检视项 | 通过条件 | 不通过条件 |
|:---|:---|:---|:---|
| 1 | 不支持类型的处理行为 | 明确说明遇到不支持类型时的具体行为（回退原生/抛出异常/跳过等） | 仅列出"不支持"但不说明行为 |
| 2 | NULL 值处理 | 明确说明输入数据包含 NULL 时的处理方式 | 未说明 |
| 3 | 空输入处理 | 明确说明空表/空分区/空文件等空输入的处理方式 | 未说明 |
| 4 | 边界值行为 | 针对关键数据类型说明边界值处理（如 INT 最大/最小值、DECIMAL 精度边界、STRING 超长字符串等） | 未说明 |
| 5 | 异常/回退触发条件 | 每个异常场景都有具体的触发条件描述，而非笼统的"某些异常情况下回退" | 触发条件模糊或缺失 |
| 6 | 多列场景行为 | 说明多列同时参与计算时的行为（如多列排序、多列 JOIN） | 未说明 |
| 7 | 数据量影响 | 说明数据量大小对行为的影响（如小数据集 vs 大数据集是否有不同路径） | 未说明 |

### 判定规则

**F8 判定**：
- 项 1（不支持类型处理行为）和项 5（异常/回退触发条件）为**关键项**，任意一个不通过 → F8 = ❌
- 其余项（2/3/4/6/7）中有 3 个及以上不通过 → F8 = ❌
- 否则 → F8 = ✅

### 检视要点提示

检视时重点关注以下信号，它们意味着行为描述不够具体：

| 信号 | 问题 |
|:---|:---|
| "不支持 XXX" 但未说明具体行为 | 测试 agent 不知道该验证回退还是异常 |
| "正常处理" 但未定义"正常" | 测试 agent 无法构造边界测试 |
| "部分支持" | 测试 agent 不知道哪些支持、哪些不支持 |
| 仅描述 happy path | 测试 agent 无法生成异常/边界用例 |
| 缺少具体触发条件 | 测试 agent 无法确定测试的输入条件 |

## F9: SQL 示例多样性（NEW）

> 此维度评估 SQL 用例示例是否覆盖了足够的场景，使下游 agent 能够据此扩展生成全面的测试用例集。
>
> **核心逻辑**：测试设计 agent 会基于 SQL 示例的模式来生成更多测试用例。如果示例只有"SELECT * FROM table"这样的单一模式，agent 无法知道需要覆盖多列组合、条件过滤、子查询等场景。多样化的示例是高质量测试生成的种子。

### 检视项

| # | 检视项 | 通过条件 | 不通过条件 |
|:---|:---|:---|:---|
| 1 | 支持类型覆盖 | SQL 示例中覆盖了文档声明的"支持"数据类型中的至少 3 种不同类型 | 仅覆盖 1-2 种类型 |
| 2 | 不支持类型回退示例 | 至少有一个 SQL 示例展示不支持类型触发回退 | 无回退示例 |
| 3 | 正常场景示例 | 至少有一个展示算子正常工作的 SQL 示例 | 无正常场景示例 |
| 4 | 边界场景示例 | 至少有一个涉及边界值的 SQL 示例（NULL、空值、极值等） | 无边界场景示例 |
| 5 | 多列/组合场景 | 至少有一个涉及多列操作的 SQL 示例（如适用） | 无多列示例 |
| 6 | 示例可执行性 | 所有 SQL 示例均为完整可执行语句（含建表、插入、查询），而非片段 | SQL 为片段或仅占位符 |

### 判定规则

**F9 判定**：
- 项 3（正常场景示例）和项 6（示例可执行性）为**关键项**，任意一个不通过 → F9 = ❌
- 其余项（1/2/4/5）中有 3 个及以上不通过 → F9 = ❌
- 否则 → F9 = ✅

### 场景覆盖对照表

检视时用此表快速判断示例覆盖情况：

| 场景类别 | 应有示例 | 检查结果 |
|:---|:---|:---|
| 单列基础类型（支持） | 每种核心类型至少 1 个 | ✅/❌ |
| 单列基础类型（不支持） | 至少 1 个展示回退 | ✅/❌ |
| NULL 值处理 | 至少 1 个含 NULL 输入 | ✅/❌ |
| 空表/空结果 | 至少 1 个空输入场景 | ✅/❌ |
| 多列操作 | 至少 1 个多列场景 | ✅/❌ |
| 复杂类型（ARRAY/MAP/STRUCT） | 声明支持时应有示例 | ✅/❌ |

---

## 总体判定规则

```
IF F1~F9 中任意功能性检视项为 ❌ THEN
  最终结论 = ❌ 不通过
ELSE
  最终结论 = ✅ 通过
END IF

非功能性问题永远不影响最终结论。
```

| 优先级 | 适用场景 | 修复要求 |
|:---|:---|:---|
