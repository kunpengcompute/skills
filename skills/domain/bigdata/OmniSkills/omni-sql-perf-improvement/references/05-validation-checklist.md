# 参考：验证清单与报告模板

> Agent trigger：代码改动完成，准备给出“是否有效”的最终结论。
>
> Agent deliverable：三方对比结果 + 正确性结论 + 风险与回退条件。

## 1. A/B 测试执行规范

### 测试矩阵

每个配置（Native / Omni-Before / Omni-After）至少跑 **3 次**，取第 2、3 次均值（排除 JIT 预热影响）。

| 配置          | 说明                              | 需固定的参数                                    |
|---------------|-----------------------------------|-------------------------------------------------|
| Native        | 关闭 Gluten，纯 Spark 原生        | `spark.gluten.enabled=false`                   |
| Omni-Before   | 优化前的 Omni 版本                | `spark.gluten.enabled=true` + 旧 jar            |
| Omni-After    | 优化后的 Omni 版本                | `spark.gluten.enabled=true` + 新 jar            |

> ⚠️ 三次测试必须使用**完全相同**的：Executor 数量 / 核数 / 内存 / 输入数据 / shuffle 分区数。

### 测试前检查

```bash
# 清除数据缓存
spark.catalog.clearCache()

# 固定 shuffle 分区（避免 AQE 导致对比不公平）
SET spark.sql.shuffle.partitions=200;
SET spark.sql.adaptive.enabled=false;  # 对比阶段建议关闭 AQE
```

## 3. 正确性验证

优化不能以牺牲正确性为代价，必须重新进行UT与端到端测试，确保sql正确性与算子不回退，测试方式查找operator-development\references\5_operator_test_guide.md，要求：
- 通过对应算子UT测试
- SQL 级别结果对比（Native vs Omni-After，行数+内容一致）

---

## 4. 稳定性检查

| 检查项                         | 方法                                              | 通过标准              |
|-------------------------------|---------------------------------------------------|-----------------------|
| 多次运行结果稳定               | 3 次结果差异 < 5%                                 | 无明显波动             |
| 长时间运行无内存泄漏           | 监控 Executor RSS 30 分钟                         | 内存平稳不增长         |
| 并发场景无竞争问题             | 多 task 并发执行相同算子                          | 结果正确，无死锁       |
| 大数据量不 OOM                 | 超出 Executor 内存上限的数据集测试                | 触发 Spill 而非 OOM   |
